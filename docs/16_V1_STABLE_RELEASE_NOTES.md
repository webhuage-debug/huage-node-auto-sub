# v1.0.1 热修复补充

v1.0.1 修复 Xray-core 内核状态识别：后端以 `/app/cores/xray/xray version` 为准判断内核是否真实可执行，并在内核管理页展示已安装状态、版本号和路径。该修复用于完成真实节点检测链路验收，不新增业务功能。

# v1.0.2 热修复补充

v1.0.2 修复 Xray 检测流程误判：软件检测链路对齐 VPS 手动成功链路，使用临时 Xray JSON 配置、`xray run -config`、本地 `127.0.0.1` SOCKS inbound，以及 `curl --socks5-hostname` 访问 `https://www.gstatic.com/generate_204`。当 curl 返回 `http_code=204` 或 `http_code=200` 时，节点判定为 `available`。

本热修复只调整检测执行链路和安全 debug 摘要，不改变订阅、领取、口令、防刷、节点池、Caddy 或 Docker 配置；不输出 raw 节点、完整 publicKey、订阅 token 或完整订阅链接。

# v1.0.3 热修复补充

v1.0.3 修复单节点检测写回问题：新增 `POST /api/detection/xray/test-node`，前端节点行按钮可对任意状态节点重新执行 Xray 检测。检测结果会写回节点池的状态、最近检测时间、响应时间、失败原因和运行时 debug，方便确认 curl exit code、HTTP 状态码和失败阶段。

本热修复同时在 Docker runner 阶段安装 `curl` 和 `ca-certificates`，确保软件检测链路依赖的 curl 可用。本版本不修改订阅、领取、口令、防刷、Caddy 或 docker-compose 配置，不提交 `data/`、`cores/` 或 `.env`。

# v1.0.4 热修复补充

v1.0.4 修复路径版单节点检测接口：`POST /api/detection/xray/test-node/:nodeId` 会按路径参数读取节点、执行 Xray 检测、写回节点池，并重新读取写回后的脱敏节点返回。该返回用于确认 `lastTestedAt` 和 `detectionRuntimeDebug` 已经真实持久化。

前端节点行按钮改为调用路径版接口。该热修复不修改 Xray 配置生成、Reality 参数解析、订阅、领取、防刷、Caddy、Dockerfile 或 docker-compose 配置，不输出 raw 节点、token 或完整订阅链接。

# v1.0.5 兜底增强补充

v1.0.5 增加手动验证可用兜底机制：管理员在 VPS 上手动验证节点可用后，可将节点标记为 `detectionCore=manual` 的可用节点，并允许其进入订阅池。该能力用于避免自动检测误判阻塞实际发布。

本版本不修改 Xray 自动检测逻辑，不修改订阅领取逻辑，不修改 Caddy、Dockerfile 或 docker-compose 配置，不提交 `data/`、`cores/` 或 `.env`。

# 16 v1.0.0 稳定版发布说明

## 文档目的

记录 `v1.0.0` 稳定版能力边界、正式发布前操作、安全边界和后续版本方向。

## 当前版本范围

`v1.0.0` 是当前稳定封版版本。本版本不新增业务功能，只完成稳定版标记、发布说明、最终验收清单和运行数据备份脚本。

## 稳定版能力清单

- GitHub 公开线索采集。
- 节点解析和本地 JSON 节点池。
- Xray-core 检测框架。
- 节点手动标记可用、不可用、恢复待检测。
- 安全订阅链接生成、重置和自动刷新。
- 订阅有效期和续期。
- 订阅二维码下载。
- 公开领取页 `/claim`。
- 视频口令验证 `/api/claim/verify`。
- 领取接口基础防刷。
- 发布前检查页。
- 发布前一键准备。
- 最终验收脚本和运行数据备份脚本。

## 当前主流程

1. 后台维护节点订阅池。
2. 后台生成安全订阅。
3. 管理员设置本期视频口令。
4. 粉丝访问公开领取页。
5. 粉丝输入视频口令。
6. 口令验证成功后复制订阅链接。

## 安全边界

- 公开域名只开放 `/claim`、`/assets/*`、`/api/claim/verify`、`/sub/*`。
- 后台 API 不公开。
- 不公开后台地址。
- 不公开 3000 端口。
- 不公开 token。
- 不公开完整订阅链接。
- 不公开真实口令。
- 不公开 raw 节点或 base64 订阅内容。

## 正式发视频前操作

1. 修改 `CLAIM_ACCESS_CODE` 为本期视频口令。
2. 执行发布前准备。
3. 下载新的二维码。
4. 测试错误口令。
5. 测试正确口令。
6. 执行最终验收脚本。
7. 确认公开域名没有开放全部 `/api/*`。

## 已知边界

- 防刷是内存级，服务重启后计数会清空。
- 节点可用性会随时间变化。
- 当前不是多活动系统。
- 当前不是多口令系统。
- 当前不做 Telegram Bot 自动分发。
- 当前不做用户登录、支付、设备数限制或渠道统计。

## 后续 v1.1 可选方向

- 多活动。
- 渠道统计。
- Telegram Bot 引导。
- 更强风控。
- 节点真实检测增强。

## 后续待补充内容

如后续进入 v1.1 开发，需要单独建立版本目标和验收清单，不能在 v1.0.0 稳定版上混入新功能。

# v1.0.6 发布管理补充

v1.0.6 在稳定版基础上增加后台发布管理页，用于发视频前集中确认本期领取口令、公开领取页、订阅状态、节点数量、有效期、生成时间和操作历史。

本版本新增运行状态文件用于保存后台设置的本期领取口令和发布操作历史。该运行状态位于 `data/` 目录，属于运行数据，不应提交到 Git。

安全边界保持不变：公开域名仍只应开放 `/claim`、`/assets/*`、`/api/claim/verify` 和 `/sub/*`；后台 API 不公开；页面不显示完整订阅 token、完整订阅链接、raw 节点或 raw/base64 内容。

# v1.1.0 自动化运行说明

v1.1.0 增加自动化运行总控。管理员开启后，系统会定时编排现有 GitHub 线索搜索、节点解析入池、Xray 检测和订阅刷新能力，减少发布前人工补池操作。

自动化默认关闭；运行状态写入 `data/automation_state.json`，该文件属于运行数据，不提交 Git。每轮日志仅保存请求次数、线索数量、新增节点数、检测节点数、可用节点数和是否刷新订阅，不保存 raw 节点、订阅 token、完整订阅链接或 GitHub Token。

安全边界保持不变：公开域名仍只开放必要路径，后台 `/api/automation/*` 不能公开到粉丝域名。
